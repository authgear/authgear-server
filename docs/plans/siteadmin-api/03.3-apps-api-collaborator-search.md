# Part 3.3: Site Admin Apps API — Collaborator Search

## Context

The existing `owner_search` parameter on `GET /api/v1/apps` only returns apps where the
**owner** matches the search keyword. This misses apps where the matching user is an editor
rather than owner — a collaborator can have the `editor` role and still be meaningfully
associated with the app.

This plan renames `owner_search` → `collaborator_search` and broadens the filter: instead of
matching only apps where `_portal_app_collaborator.role = 'owner'` and the user ID is in the
search result, we now match any app where **any collaborator (any role)** has a user ID in the
search result.

`owner_search_truncated` is similarly renamed to `collaborator_search_truncated`.

All other behaviour (search keyword resolution via Admin API, truncation cap of 20 users,
`relevance` sort, `owner_search_truncated` semantics, sort/order interactions) is unchanged.

---

## Limitations and considerations (inherited from 03.2)

All limitations from 03.2 still apply, with `owner_search` → `collaborator_search` in the
wording:

| # | Limitation | Impact |
|---|---|---|
| 1 | **Hard cap of 20 user IDs** | `collaboratorSearchPageSize = 20`. A keyword matching >20 users truncates to the top 20. |
| 2 | **`total_count` is a lower bound when truncated** | Only apps whose collaborators are among the top-20 matched users are counted. |
| 3 | **Seq scan on `_portal_app_collaborator`** | The EXISTS subquery scans the table by `app_id`. No new index is needed beyond the existing composite `UNIQUE(app_id, user_id)`. |
| 4 | **Collaborator-rank dominates pagination** | When `sort=relevance`, `MIN(array_position)` is the primary ORDER BY. Apps whose best-matching collaborator has a high rank fill earlier pages. |
| 5 | **Best-effort for broad queries** | Searching `"a"` or `"@gmail.com"` still truncates. |
| 6 | **Extra HTTP round-trip** | `collaborator_search` still calls Admin API before the SQL query. |

## SQL changes

### Old filter (owner-only)

```sql
-- `ac` is joined with `AND ac.role = 'owner'`
WHERE ac.user_id = ANY(?)
```

This matched only apps whose **owner** was in the search result.

### New filter (any collaborator, any role)

```sql
WHERE EXISTS (
    SELECT 1 FROM _portal_app_collaborator acf
    WHERE acf.app_id = cs.app_id AND acf.user_id = ANY(?)
)
```

Uses an `EXISTS` subquery so the owner-only `LEFT JOIN ac` (kept for `owner_user_id` display)
does not restrict the filter.

### Old relevance ORDER BY

```sql
-- rank by position of the owner in the search-result array
array_position(ARRAY['uid1','uid2',...]::text[], ac.user_id) ASC
```

### New relevance ORDER BY

```sql
-- rank by the BEST (lowest) position among matching collaborators of the app
(SELECT MIN(array_position(ARRAY['uid1','uid2',...]::text[], acr.user_id))
 FROM _portal_app_collaborator acr
 WHERE acr.app_id = cs.app_id AND acr.user_id = ANY(?)) ASC
```

`MIN` is over only the collaborators who are in the search-result array.
The `AND acr.user_id = ANY(?)` filter lets PostgreSQL use the `UNIQUE(app_id, user_id)` index
for point lookups rather than scanning all collaborators of the app.
Since the `EXISTS` filter guarantees at least one matching collaborator per result row,
`MIN(...)` is never `NULL`.

## API spec changes

### `GET /api/v1/apps` — parameter changes

- **Rename** `owner_search` → `collaborator_search`.
- **Update description**: searches all collaborators (any role), not just owners.

```yaml
- name: collaborator_search
  in: query
  description: >
    Filter by collaborator keyword. Searches the portal user database by partial
    email, name, or other attributes. Returns apps that have any collaborator
    (owner or editor) whose profile matches the keyword. When provided and `sort`
    is omitted, defaults to `sort=relevance`. At most 20 matching users are
    considered; if more than 20 users match, `collaborator_search_truncated` is
    set to `true` in the response and `total_count` is a lower bound.
  schema:
    type: string
```

**Updated `sort` description** (replace `owner_search` references):

```yaml
- name: sort
  in: query
  description: >
    Field to sort by. Defaults to `created_at`, or `relevance` when
    `collaborator_search` is provided. `relevance` sorts by how well the app's
    best-matching collaborator matches the `collaborator_search` keyword and
    requires `collaborator_search` to be set; using `relevance` without
    `collaborator_search` returns 400.
```

### `AppsListResponse` schema — rename field

```yaml
AppsListResponse:
  properties:
    collaborator_search_truncated:   # was owner_search_truncated
      type: boolean
      description: >
        True when the `collaborator_search` keyword matched more than 20 portal
        users and results are limited to the top 20 matches. `total_count` is a
        lower bound in this case. Always `false` when `collaborator_search` is
        not provided.
```

### `ListAppsWithStats` — SQL changes

**Filter** (inside `withFilters`):

```go
if len(params.CollaboratorUserIDs) > 0 {
    q = q.Where(
        "EXISTS (SELECT 1 FROM "+collaboratorTable+" acf WHERE acf.app_id = cs.app_id AND acf.user_id = ANY(?))",
        pq.Array(params.CollaboratorUserIDs),
    )
}
```

**Relevance ORDER BY**:

```go
if params.Sort == siteadmin.Relevance && len(params.CollaboratorUserIDs) > 0 {
    quotedIDs := make([]string, len(params.CollaboratorUserIDs))
    for i, id := range params.CollaboratorUserIDs {
        quotedIDs[i] = "'" + id + "'"
    }
    inlinedArray := "ARRAY[" + strings.Join(quotedIDs, ",") + "]::text[]"
    rankExpr := "(SELECT MIN(array_position(" + inlinedArray + ", acr.user_id))" +
        " FROM " + collaboratorTable + " acr" +
        " WHERE acr.app_id = cs.app_id AND acr.user_id = ANY(" + inlinedArray + "))"
    orderExpr = rankExpr + " ASC, " + orderExpr
}
```

> **Note:** `AND acr.user_id = ANY(...)` is added to the ranking subquery so PostgreSQL can use the
> `UNIQUE(app_id, user_id)` index for point lookups instead of scanning every collaborator of the
> app and discarding non-matching rows via `MIN`'s NULL-ignore behaviour. Without this filter the
> subquery would still produce correct results (NULLs are ignored by MIN) but would do unnecessary
> work proportional to the total number of collaborators per app rather than the search-set size.
> The array is inlined as a SQL literal (safe for UUIDs which contain only `[0-9a-f-]`) because
> squirrel's `OrderBy()` does not support bound parameters.

### `AppService.ListApps` — updated flow

Only naming changes; the logic is identical to 03.2:

1. Normalise sort: if `CollaboratorSearch != ""` and sort omitted → default `Relevance`
2. Resolve `CollaboratorSearch` → `[]collaboratorUserIDs` via `AdminAPI.SearchUsersByKeyword`
3. Compute `lastMonthStart`
4. DB query via `OwnerStore.ListAppsWithStats` with `CollaboratorUserIDs`
5. Resolve owner emails for the current page
6. Return `CollaboratorSearchTruncated`

Error message update: `"sort=relevance requires owner_search to be set"` →
`"sort=relevance requires collaborator_search to be set"`

### `fakeOwnerStore` — add `collaborators` map

```go
type fakeOwnerStore struct {
    rows          []AppStoreRow
    owners        map[string]string
    // collaborators maps appID → all collaborator user IDs (all roles).
    // When set, overrides OwnerUserID for filter matching and relevance ranking.
    // When not set for an app, falls back to OwnerUserID only.
    collaborators map[string][]string
}
```

**Filter update** (inside `ListAppsWithStats`):

```go
if len(params.CollaboratorUserIDs) > 0 {
    matched := false
    if collabs, ok := f.collaborators[r.AppID]; ok {
        for _, uid := range collabs {
            if slices.Contains(params.CollaboratorUserIDs, uid) {
                matched = true
                break
            }
        }
    } else {
        matched = slices.Contains(params.CollaboratorUserIDs, r.OwnerUserID)
    }
    if !matched {
        continue
    }
}
```

**Relevance rank helper**:

```go
func (f *fakeOwnerStore) bestRankFor(row AppStoreRow, collaboratorUserIDs []string) int {
    sentinel := len(collaboratorUserIDs)
    if collabs, ok := f.collaborators[row.AppID]; ok {
        best := sentinel
        for _, uid := range collabs {
            if pos := slices.Index(collaboratorUserIDs, uid); pos >= 0 && pos < best {
                best = pos
            }
        }
        return best
    }
    pos := slices.Index(collaboratorUserIDs, row.OwnerUserID)
    if pos < 0 {
        return sentinel
    }
    return pos
}
```

Replace the `ownerRank` map in the sort with `f.bestRankFor(a, params.CollaboratorUserIDs)`.

`bestRankFor` already iterates only over collaborators that appear in `collaboratorUserIDs`,
mirroring the `AND acr.user_id = ANY(?)` filter in the SQL ranking subquery.
